# ドキュメント実装計画

## 目的
全画面のワイヤーフレームHTMLファイルおよび共通コンポーネントの機能概要・一覧ドキュメントを作成する。
`message-search.md` をテンプレートとして、品質の統一を図りつつ、並列作業可能なようにタスクを分割する。

## 方針
- **並列実行**: 画面を機能群（グループ）ごとに分け、並列してドキュメント作成を進められるようにする。
- **標準化**: `docs/design/仕様書/message-search.md` のフォーマット（EARS記法、機能一覧など）を厳守する。
- **共通コンポーネント**: 各画面のドキュメント作成時に、使用している共通コンポーネントを特定し、`docs/design/仕様書/共通コンポーネント一覧.md` を更新する。共通コンポーネント自体の詳細ドキュメントも別途作成する。
- **実装タスク計画**: 実装タスク計画を立ててからドキュメント作成に取り掛かること。

## ドキュメント作成ルール
1. **格納場所**: `docs/design/仕様書/[グループ名]/`
   - 例: `docs/design/仕様書/パブリック・トップ・認証/`
2. **ファイル名**: `画面名(日本語).md` または `共通コンポーネント名(日本語).md`
3. **言語**: 日本語
4. **必須項目**:
    - 機能の目的
    - ユースケース
    - EARS記法による要件定義
    - 機能一覧
    - ユーザ確認事項
5. **共通コンポーネント一覧の更新**:
    - 新たに共通コンポーネントのドキュメントを作成した場合、`共通コンポーネント一覧.md` の「ドキュメントへのリンク」を更新する。

## 進め方
1. 各グループの担当者は、対象のHTMLファイルを解析する。
2. `message-search.md` を参考にドキュメントを作成する。
3. 共通コンポーネントの詳細な挙動が必要な場合は、コンポーネントのソースコードを確認し、必要に応じてコンポーネントのドキュメントも作成する。
4. 作成したドキュメントを `docs/design/仕様書/` に保存する。
5. ユーザレビューを依頼する。
